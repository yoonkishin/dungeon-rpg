# Growth / Promotion V1 Plan

`dungeon-rpg`에 `lightsaber_test` 문서 감각을 참고한 **성장/전직 1차 적용안**.

> 구현 현황 메모 (2026-04-10)
> - Phase A 완료: XP 커브, infantry 성장 라인, save/load 호환 적용
> - Phase B 완료: HUD/프로필 클래스명 표시 정리
> - Phase C 완료: 수련의 방 패널과 수동 승급 확정 플로우 적용

---

## 1. 목표

현재 `dungeon-rpg`는 레벨업과 티어가 존재하지만,
실제 구조는 아직 다음에 가깝다.

- 레벨업 = 고정 능력치 상승
- 티어 = 레벨 구간별 외형/연출 변화
- 성장 축 = 사실상 `level` 하나

이번 1차의 목표는 이를 아래처럼 바꾸는 것이다.

- **레벨**: 전투/퀘스트 보상으로 오르는 기본 성장축
- **병종 라인**: 보병/비병/기병/수병/창병/궁병/승려/신관/법사/사교
- **전직(승급)**: 특정 레벨 구간에서 라인별 이름/표시/능력 보정이 바뀌는 구조
- **장비**: 기존 장비 보정은 유지하되, 성장/전직 구조와 UI 해석을 맞춘다

즉,
`추상 tier 연출` 중심 구조에서
`병종 기반 성장/승급 해석` 중심 구조로 이동한다.

---

## 2. 이번 1차에서 반드시 할 것

### 2.1 플레이어 성장 곡선 교체

현재:
- `xpNext *= 1.4`
- Lv35 cap
- 레벨업마다 HP/MP/ATK/DEF 고정 증가

변경:
- `lightsaber_test` 성장 문서의 **2차 곡선 감각**을 참고한 XP 커브 적용
- Lv35 cap은 유지
- 레벨업 시 증가 능력치는 **병종 라인 기반 성장 테이블**을 참조

### 2.2 플레이어 라인(병종) 고정

프로토타입에서는 플레이어 시작 라인을 1개 고정한다.

권장 기본값:
- 시작 라인: `Infantry` (보병)

이유:
- 현재 장비/공격 구조와 가장 잘 맞음
- 시작 캐릭터 해석이 쉽다
- 이후 다른 라인 실험 확장도 가능하다

### 2.3 추상 티어를 라인 승급 이름으로 교체

현재 `CLASS_TIERS`:
- 수련생 / 모험가 / 전사 / 정예 전사 / 영웅 ...

변경 방향:
- 보병 라인 기준 예시:
  - Lv1~5: 견습보병
  - Lv6~10: 글라디에이터
  - Lv11~15: 소드맨
  - Lv16~20: 파이터
  - Lv21~25: 로열파이터
  - Lv26~30: 제네럴
  - Lv31~35: 로열가드

중요:
- 이번 단계는 **실제 다중 전직 분기 구현이 아니라**,
  `현재 레벨 구간을 어떤 클래스명으로 읽을지`를 교체하는 단계다.

### 2.4 수련의 방(Training Room) 1차 도입

현재 마을에는 신전/상점/장신구/현자 등이 있다.

추가 방향:
- `수련의 방` NPC 또는 패널 추가
- 현재 레벨이 승급 구간에 도달했을 때,
  플레이어가 **승급 확인 UI**를 보고 확정하는 흐름 추가

이번 단계에서 수련의 방이 하는 일:
- 다음 승급 가능 여부 안내
- 현재 라인의 다음 클래스명 표시
- 승급 시 바뀌는 핵심 보너스 표시
- 확정 시 `player.classRank` 갱신

### 2.5 동료 성장 해석 정리

동료는 이미 `classId`와 `COMPANION_CLASS_PROFILES`를 가진다.

이번 단계에서는:
- 동료 개별 레벨업은 아직 하지 않음
- 대신 동료 카드에
  - 병종명
  - 역할명
  - 기본 AI 성향
  - 현재 운영 모드
  를 더 선명하게 표시

즉, 동료는 이번 1차에서 **전직 구현 대상이 아니라, 병종 정체성 표시 강화 대상**으로 본다.

## 3. 이번 1차에서 하지 않을 것

아래는 일부러 미룬다.

- 13축 전체 스탯 도입
- 능력포인트 / 마법포인트 분배 UI
- 문장 시스템
- 8~10단 마스터/그랑/그랜드 융합 라인
- 플레이어 라인 선택/변경 시스템
- 동료 개별 레벨업 및 승급
- 용병 실제 고용 시스템

핵심은:
**성장 해석을 바꾸되, 프로토타입의 속도를 해치지 않는다.**

## 4. 데이터 구조 제안

## 4.1 PlayerGrowthLine

`data.js` 또는 별도 성장 섹션에 추가.

```js
const PLAYER_GROWTH_LINES = {
  infantry: {
    lineId: 'infantry',
    unitType: 'Infantry',
    lineName: '보병',
    ranks: [
      { rank: 1, className: '견습보병', reqLevel: 1, statBonus: { maxHp: 0, maxMp: 0, atk: 0, def: 0, speed: 0 } },
      { rank: 2, className: '글라디에이터', reqLevel: 6, statBonus: { maxHp: 10, atk: 2, def: 1 } },
      { rank: 3, className: '소드맨', reqLevel: 11, statBonus: { maxHp: 18, atk: 4, def: 1 } },
      ...
    ],
    levelGrowth: {
      maxHp: 18,
      maxMp: 8,
      atk: 4,
      def: 1,
      speed: 0.03,
      critChance: 0.5,
    }
  }
};
```

## 4.2 Player runtime state 추가

`state.js`의 player에 아래 필드 추가 권장:

```js
classLine: 'infantry',
classRank: 1,
promotionPending: false,
```

의미:
- `classLine`: 현재 병종 라인
- `classRank`: 현재 승급 단계
- `promotionPending`: 수련의 방에서 확정 가능한 승급이 있는지

## 5. UI 적용 포인트

### 5.1 프로필 패널

현재 프로필은:
- tier 숫자
- tier 이름
- 다음 티어 레벨

변경:
- `n단 - 클래스명` 표시
- 병종 라인 표시 (`보병 라인`)
- 다음 승급명 표시
- 수련의 방 안내 문구 표시

예시:
- `3단 - 소드맨`
- `라인: 보병`
- `다음 승급: 파이터 (Lv.16)`

### 5.2 승급 배너

현재 `showTierBanner()`는 단순히 tier 이름만 띄운다.

변경:
- `승급! 소드맨`
- 가능하면 색상은 라인 유지, 명칭만 교체

### 5.3 수련의 방 패널

간단 UI면 충분하다.

필요 정보:
- 현재 클래스명
- 현재 레벨
- 다음 승급명 / 필요 레벨
- 승급 시 보너스
- `[승급]` 버튼

## 6. 구현 순서

### Phase A — 성장 테이블 이식
- XP 곡선 함수 추가
- 병종 라인 1개(`infantry`)만 정의
- 레벨업 능력치 증가를 테이블 기반으로 교체
- 기존 티어 연출이 깨지지 않게 최소 수정

### Phase B — 클래스명/승급 표시 교체
- `CLASS_TIERS` 중심 UI를 `classLine/classRank` 중심으로 교체
- 프로필 / HUD / 배너 문구 정리

### Phase C — 수련의 방 1차
- NPC 추가
- 승급 가능 시 패널 오픈
- 수동 승급 확정 흐름 추가
- save/load 연동

---

## 7. 저장 데이터 변경

`save.js`에 아래 필드 추가 필요:

- `player.classLine`
- `player.classRank`
- `player.promotionPending`

세이브 호환 원칙:
- 기존 세이브는 `classLine='infantry'`, `classRank=getRankFromLevel(player.level)`로 복구

---

## 8. 완료 기준

이번 1차는 아래가 되면 완료로 본다.

1. 플레이어 성장 커브가 더 이상 단순 `xpNext *= 1.4`가 아니다.
2. 플레이어는 `수련생/영웅` 같은 추상 티어 대신, 병종 라인의 클래스명으로 표시된다.
3. 프로필 패널에서 현재 클래스 / 다음 승급을 읽을 수 있다.
4. 수련의 방에서 승급 가능 여부를 확인할 수 있다.
5. 기존 세이브가 깨지지 않는다.
6. 동료 시스템/상점/UI 리팩토링과 충돌하지 않는다.

---

## 9. 한 줄 결론

이번 단계의 핵심은
**전투 프로토타입에 “원작풍 성장 해석”을 입히는 것**이지,
원작 전체 성장 시스템을 한 번에 다 옮기는 게 아니다.
