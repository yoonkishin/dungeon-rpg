# Repo Strategy - lightsaber_test vs dungeon-rpg

이 문서는 두 저장소를 어떻게 병행 운영할지 정리한다.

- `lightsaber_test` = 메인 개발 저장소
- `dungeon-rpg` = 프로토타입 저장소

둘은 연결되어 있지만, 같은 역할을 하지 않는다.

---

## 1. 역할 분리

### lightsaber_test
메인 제품 개발용 저장소.

특징:
- Unity / C# 중심
- 문서 기준 작업
- 구조/이름/데이터 기준 엄격
- 장기 유지보수 고려
- 실제 제품으로 성장시키는 저장소

### dungeon-rpg
빠른 검증용 저장소.

특징:
- 웹 기반 프로토타입
- 빠른 실행 / 빠른 실험 우선
- 구조보다 체감 우선
- 실패 비용 낮게 유지
- 아이디어 검증용

---

## 2. 병행 운영 원칙

두 repo를 동시에 운영해도 괜찮다.
다만 아래를 지킨다.

1. 같은 목적이라고 생각하지 않는다.
2. 같은 구조를 강제로 맞추지 않는다.
3. 프로토타입 결과를 메인에 옮길 때는 재설계한다.
4. repo 별로 커밋 맥락을 분리한다.
5. repo 별 규칙 문서를 각각 둔다.

---

## 3. 무엇을 어디서 개발할지

### dungeon-rpg 에서 할 것
- 전투감 테스트
- 조작감 테스트
- 모바일 UI 배치 테스트
- 미니맵/전환/저장 UX 실험
- 빠른 적 AI 패턴 실험
- “재밌는가?” 확인

### lightsaber_test 에서 할 것
- 실제 게임 구조 설계
- 씬 흐름 고정
- 데이터 구조 설계
- ScriptableObject 기준 확립
- 장기 운영 가능한 코드 작성
- 문서와 구현 동기화

---

## 4. 메인에 반영하는 방법

프로토타입에서 메인으로 옮길 때는 아래 순서를 따른다.

1. 무엇이 좋았는지 한 문장으로 정리
2. 왜 좋았는지 UX/전투 관점으로 설명
3. 메인 구조에 맞게 다시 설계
4. 필요하면 문서 먼저 수정
5. 그 다음 구현

즉, **코드를 복붙하지 말고 결론을 이식**한다.

---

## 5. 절대 섞지 말아야 할 것

- repo 간 브랜치 혼용
- 현재 디렉토리 착각한 커밋/푸시
- 실험 코드를 메인 기준처럼 확정
- 메인 규칙을 프로토타입에 과하게 강제
- 프로토타입의 임시 수치를 메인 canonical 값처럼 사용

---

## 6. 커밋/브랜치 권장 규칙

### lightsaber_test
권장 접두어:
- `docs:`
- `feat:`
- `refactor:`
- `fix:`
- `test:`

### dungeon-rpg
권장 접두어:
- `proto:`
- `tune:`
- `experiment:`
- `fix:`
- `docs:`

이렇게 나누면 히스토리만 봐도 repo 성격이 안 섞인다.

---

## 7. 최종 기준

의사결정이 헷갈리면 이렇게 판단한다.

- **지금 빨리 체감 확인이 필요한가?** → `dungeon-rpg`
- **이 결정이 메인 제품 구조에 들어갈 것인가?** → `lightsaber_test`
- **아이디어를 검증 중인가, 제품으로 축적 중인가?**
  - 검증 중 → `dungeon-rpg`
  - 축적 중 → `lightsaber_test`

---

## 8. 한 줄 결론

`dungeon-rpg`는 실험실이고,
`lightsaber_test`는 본선이다.

실험실에서 배운 걸 본선에 가져오되,
둘을 같은 경기장처럼 운영하지 않는다.
