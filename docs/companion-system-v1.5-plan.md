# Companion System Plan v1.5

이 문서는 `dungeon-rpg`의 현재 동료 시스템을
`lightsaber_test`의 파티/동료 설계 감각을 참고해
**프로토타입답게 빠르게 유지하면서도, 메인 저장소로 인사이트를 옮기기 쉬운 형태**로 재정리하기 위한 짧은 기준 문서다.

> 상태 업데이트 (2026-04-10): 이 문서의 핵심 방향은 1차 구현까지 이미 반영됐다. 현재 코드는 `class-based 10병종 roster`, `CompanionClassProfile`, `AI 모드`, `동료 UI 정리`, `state.js` 분리를 포함하고 있으며, 이 문서는 앞으로 구현 전 spec보다는 설계 의도와 후속 정리 기준 문서로 본다.

---

## 1. 목표

현재 `dungeon-rpg`의 동료 시스템은 이미 다음을 갖고 있다.

- 활성 동료 2명 제한
- 동료별 역할 차이
- 동료별 고유 스킬
- 조합 시너지
- 사망 / 부활 / 선택 UI

하지만 지금 구조는 아직
**"동료 = 전투 프리셋"** 성격이 강하다.

이번 계획의 목표는 이를

- **캐릭터성**
- **전투 역할**
- **AI 지시 모드**
- **조합 보너스**

로 분리해서,
**프로토타입은 계속 빠르게 실험하되 메인 제품 설계로 옮길 때 재사용 가능한 판단 구조**를 만드는 것이다.

---

## 2. 고정 기준

프로토타입 기준은 아래로 고정한다.

- 플레이어 1명 + 활성 동료 2명
- 현재 플레이 루프는 **3인 파티 고정**으로 검증한다
- 다만 구조적으로는 **용병 1명 슬롯을 미래 확장용으로 예약**한다
- 던전 내부에서는 동료 교체 불가
- 마을/필드에서만 동료 교체 가능
- 동료는 자동 전투 기반
- 플레이어는 동료의 세부 조작이 아니라 **행동 방침**을 지정한다

즉, 지금은 `지휘관 1 + 동료 2`를 핵심 시스템으로 검증하되,
데이터 구조와 UI 개념에는 `+ 용병 1` 확장 여지를 남긴다.

이 기준은 모바일 화면 밀도와 현재 구현 상태를 고려했을 때 가장 안정적이다.

---

## 3. 메인 설계에서 참고할 핵심

`lightsaber_test`에서 참고할 핵심은 다음이다.

1. 플레이어 + AI 동료 중심 파티 감각
2. 동료별 전투 역할 구분
3. 파티 관리 화면의 필요성
4. AI 행동 모드 (Aggressive / Defensive / Support)
5. 던전 내부 교체 제한
6. 문서에 정리된 **10개 캐릭터/병종 계열 정보**를 동료 정체성 설계의 기준으로 참고한다

특히 프로토타입 동료는 단순히 "던전 보상 유닛"이 아니라,
`lightsaber_test`에 정리된 계열 감각을 축약 반영한 **이름 있는 캐릭터**로 해석한다.

단, 프로토타입 repo에서는 이를 제품 수준 구조로 과설계하지 않는다.

즉, **코드를 옮기는 게 아니라 구조적 판단을 가져온다.**

## 4. 구조 재정리 원칙

현재 동료 시스템은 아래 4층으로 나눠서 본다.

### 4.1 CompanionCharacter
"누구인가"를 담당한다.

예시 속성:
- companionId
- 이름
- 획득/합류 조건
- 설명
- 기본 테마 색
- 대표 이미지/초상 개념
- 계열 키 (`lineageKey`)

예:
- 슬라임 기사
- 고블린 궁수
- 해골 마법사
- 성녀

### 4.2 CompanionCombatProfile
"전투에서 어떻게 싸우는가"를 담당한다.

예시 속성:
- roleKey
- roleLabel
- attackRange
- preferredRange
- attackCooldown
- skillId
- skillName
- 전투 성향 설명

현재 `COMPANION_PROFILES`는 이 계층에 가깝다.

### 4.3 CompanionAIMode
"지금 어떤 방식으로 행동하라고 지시받았는가"를 담당한다.

모드:
- `Aggressive`
- `Defensive`
- `Support`

중요한 점:
- `roleKey`는 캐릭터의 기본 성향
- `aiMode`는 플레이어가 주는 전투 운영 지시

즉 같은 원거리 캐릭터라도
- Aggressive면 더 적극적으로 압박하고
- Defensive면 플레이어 근처를 더 지키고
- Support면 생존/보조를 우선할 수 있다.

### 4.4 PartySynergy
"조합이 주는 추가 보너스"를 담당한다.

현재 시너지 시스템은 유지하되,
이것을 핵심 본체가 아니라 **중간 강도의 보조 레이어**로 둔다.

추천 구조는 2층이다.

- **역할 커버 보너스**: 탱커+원거리, 탱커+서포터처럼 조합의 균형이 맞을 때 주는 범용 보너스
- **캐릭터 페어 시너지**: 특정 캐릭터 둘을 함께 편성했을 때 주는 고유 보너스

원칙:
- 시너지는 있으면 편성 이유가 생기지만, 없어도 파티가 망가지지 않아야 한다
- 캐릭터 애착과 역할 선택이 먼저이고, 시너지는 그 위에 얹는 보너스다
- 미래에 추가될 **용병 슬롯은 역할 커버 보너스만 받고, 캐릭터 페어 시너지에서는 제외**하는 것을 기본값으로 본다

즉,
- 캐릭터가 본체
- 역할이 전투 해석
- AI 모드는 플레이어 지시
- 시너지는 추가 보너스

이라는 순서를 유지한다.

## 5. 권장 데이터 모델

프로토타입에서는 아래처럼 단순하게 나누는 것을 권장한다.

```js
PartyFormation = {
  commanderSlot,
  companionSlotA,
  companionSlotB,
  mercenarySlot, // reserved for future
}

CompanionCharacter = {
  id,
  name,
  lineageKey,
  unlockRule,
  description,
  portraitKey,
}

CompanionCombatProfile = {
  companionId,
  roleKey,
  roleLabel,
  attackRange,
  preferredRange,
  attackCooldown,
  skillId,
  skillName,
  skillCooldown,
}

CompanionRuntimeState = {
  companionId,
  hp,
  maxHp,
  attackTimer,
  skillTimer,
  flashTimer,
  aiMode,
}
```

여기서 `lineageKey`는 `lightsaber_test`의 병종/캐릭터 계열 감각을 축약 반영하기 위한 키다.
예: `infantry`, `archer`, `monk`, `priest`, `mage` 등.

이 구조는 지금 JS 전역 상태 스타일을 유지하면서도,
나중에 Unity/ScriptableObject 기반으로 옮길 때 매핑이 쉬워진다.

## 6. AI 모드 설계 기준

### Aggressive
- 더 멀리 적을 추적
- 스킬 사용을 더 빠르게 시도
- 저체력 적 마무리와 압박 우선
- 플레이어와 거리가 벌어져도 일정 수준까지 추격 허용
- **플레이어가 바로 체감할 정도로 공격성이 높아야 한다**

### Defensive
- 플레이어 근처 유지 성향 강화
- 너무 멀리 추격하지 않음
- 체력/거리 기준으로 이탈 빈도 증가
- 전선 유지와 호위 우선
- **위험 회피와 자리 유지가 눈에 띄어야 한다**

### Support
- 후방 유지
- 힐 / 보호 / 군중제어 우선
- 무리한 추격 최소화
- 직접 킬보다 유지력 우선
- **서포트형 동료는 모드 전환 시 운영 방식이 분명히 달라 보여야 한다**

### 해석 원칙
AI 모드는 `roleKey`를 덮어쓰는 개념이 아니라,
**기본 역할 위에 행동 가중치를 얹는 방식**으로 해석한다.

예:
- 탱커 + Defensive = 플레이어 주변 전선 유지 강화
- 원거리 + Aggressive = 사거리 활용 압박 강화
- 서포터 + Support = 힐/보호 우선도 증가

## 7. UX 기준

동료 패널은 단순 선택창이 아니라
**파티 관리창**에 가까워져야 한다.

패널에서 최소한 보여야 하는 것:
- 현재 활성 동료 2명
- 대기 동료 목록
- 미래 확장용 용병 슬롯의 존재 또는 잠금 상태
- 각 동료 역할
- 고유 스킬 이름
- 현재 HP 상태
- 현재 AI 모드
- 현재 조합 시너지
- 교체 가능 여부

표시 규칙:
- 던전 내부: 교체 불가
- 마을/필드: 교체 가능
- 사망 상태: 명확히 표시
- 부활 가능 시 비용 표시

UI 목표는 화려함이 아니라
**동료를 보고 바로 편성 결정을 내릴 수 있게 하는 것**이다.

## 8. 합류/성장 해석 기준

프로토타입에서는 모든 동료를 플레이어 본체처럼 깊게 육성시킬 필요는 없다.

추천 방향은 **가벼운 성장 + 뚜렷한 역할 차이 + 단계 해금**이다.

### 8.1 핵심 동료
메인 설계 감각을 검증할 대표 동료 3명 정도를 둔다.

권장 예시:
- 초반 탱커 1명
- 중반 원거리 1명
- 후반 서포터 1명

이 축은 `lightsaber_test` 문서의 계열/캐릭터 정보를 참고해,
동료가 단순 유닛이 아니라 **이름 있는 핵심 캐릭터**로 느껴지게 만든다.

### 8.2 확장 동료
나머지는 실험용 역할/조합 검증용으로 유지한다.

즉,
- 핵심 동료 = 메인 설계 감각 검증
- 확장 동료 = 프로토타입 재미/조합 실험

### 8.3 성장 원칙
- 동료는 지휘관과 완전히 분리된 복잡 육성 대신 **레벨 동기화 기반**으로 운용한다
- 성장의 핵심은 수치 파밍보다 **역할 특화 강화**다
- 성장 체감은 HP/ATK 소폭 상승보다, 행동 변화/보조 능력 강화/운용 차이에서 오게 한다
- 초기 단계에서는 동료 장비 시스템을 크게 열지 않는다

### 8.4 승급 원칙
- 초반에는 다단 전직을 전부 구현하지 않는다
- 대신 특정 레벨 또는 진행 마일스톤에서 **1차 승급** 정도만 검토한다
- 승급은 단순 수치 증가보다, 역할 정체성이 더 선명해지는 방향으로 설계한다

예:
- 탱커: 보호 범위 증가 / 경직 강화
- 원거리: 사거리 / 마무리 능력 강화
- 서포터: 힐 임계치 / 버프 유지력 강화

## 9. 구현 우선순위 / 반영 상태

### 1순위: AI 모드 추가 (반영 완료)
이유:
- `lightsaber_test` 감각을 가장 직접적으로 반영함
- 체감 차이가 큼
- 프로토타입 가치가 바로 올라감

### 2순위: 동료 데이터 구조 분리 (반영 완료)
이유:
- 프로토타입이 커지기 전에 구조를 정돈해야 함
- 메인 repo로 판단을 옮기기 쉬워짐

### 3순위: 파티 관리 UI 개선 (1차 반영 완료)
이유:
- AI 모드와 시너지 정보를 플레이어가 이해해야 함
- 편성 UX가 좋아져야 시스템 검증이 가능함

### 4순위: 합류/성장 규칙 다듬기 (후속 여지)
이유:
- 시스템 골격이 먼저 안정돼야 콘텐츠 규칙이 의미가 생김

## 10. 이번 계획의 완료 기준

아래가 모두 만족되면 이번 재정리 방향은 성공으로 본다.

1. 동료 시스템이 캐릭터 / 역할 / AI모드 / 시너지 / 성장 원칙으로 설명 가능하다.
2. 플레이어가 동료별 AI 방침을 바꿀 수 있고, 모드 차이가 실제로 체감된다.
3. 파티 관리 화면에서 편성 판단이 쉬워진다.
4. 현재 프로토타입의 속도와 단순함이 크게 망가지지 않는다.
5. `지휘관 1 + 동료 2` 구조를 선명하게 검증하면서도, 미래의 `+ 용병 1` 확장 여지를 유지한다.
6. `lightsaber_test`로 옮길 때 "무엇을 배웠는지"를 한 문장으로 정리할 수 있다.

구조는 `lightsaber_test`의 파티/동료 설계로 승격하기 쉬운 방향으로 정리한다.
